# 统一交付执行标签布局设计

## 背景

Application 构建详情当前先展示 Build Pipeline 步骤和日志，再以独立大区块展示环境发布、
Deploy-only Pipeline 步骤和日志。两个区块重复标题、步骤导航和日志容器，页面纵向过长，切换环境时
视线需要在上下区块间移动。

## 目标

将同一构建版本的构建执行与各环境部署执行合并为一组同级执行标签。用户选择“构建”、QA、PROD
等标签后，下方同一个步骤与日志区域原位更新，使构建与环境发布的关系更直接、页面更紧凑。

## 布局

- 构建信息区下方增加统一的“交付执行”区域。
- 第一行展示执行标签：“构建”固定在首位，随后按发布批次 target 顺序展示环境。
- 每个标签包含名称和当前状态；选中项使用现有主色边框，不使用大尺寸环境卡片。
- 第二部分只保留一个步骤标签区域和一个日志面板，根据所选执行对象展示 Build 或 Deploy-only
  PipelineRun 的实际 Task/Step。
- 移除独立的“环境发布”和“PROD 部署步骤”等重复区块标题。
- 宽屏标签自适应排列；窄屏保持单行横向滚动，避免把每个环境拉成整行卡片。

## 交互与状态

- 页面默认选中“构建”，保持用户进入详情后先看到构建过程。
- 点击有 PipelineRun 的环境标签，加载并展示该 Deploy-only PipelineRun 的实际步骤和日志。
- 等待审批、等待构建或尚未创建 PipelineRun 的环境可以选中；下方显示对应真实等待说明，不发起日志请求。
- 切换构建时恢复选中“构建”，清空旧环境日志。
- 切换执行标签时继续使用独立请求代次保护，旧日志响应不能覆盖当前选择。
- 构建或任一环境处于活跃状态时维持现有轮询；刷新后保留当前环境选择（只要该 target 仍存在）。

## 组件边界

- 新建或重构统一执行面板组件，负责执行标签、当前步骤和日志的组合展示。
- `ApplicationBuildDetail` 只负责构建元数据和组合统一执行面板。
- `ApplicationBuildExplorer` 继续负责 API 数据、选择状态、日志请求和竞态保护，不在 View 中新增 HTTP 请求。
- 复用 `PipelineStepLogPanel` 的步骤与日志呈现能力，并允许隐藏重复的区块标题。

## 错误与空状态

- 构建没有 PipelineRun 时显示“此构建没有关联 PipelineRun”。
- 环境没有 PipelineRun 时根据 target 状态显示等待审批、等待构建或等待创建部署 Pipeline。
- 日志读取失败只影响当前执行对象的日志区，不影响其他执行标签及构建元数据。
- 没有环境 target 时只展示“构建”标签，不额外显示空的环境发布区块。

## 验证

- 状态测试覆盖默认选中构建、环境选择保持、target 消失后的回退和等待态不请求日志。
- 组件或渲染测试覆盖统一标签顺序、只存在一个步骤日志区域，以及构建/环境切换内容。
- 运行前端测试、Vue 类型检查、生产构建和 `./scripts/verify.sh`。

## 非目标

- 不修改 Build/Deploy Pipeline、发布批次、审批或后端 API。
- 不在此页面增加批准、重试或追加环境操作。
- 不合并构建与部署的执行数据；仅统一查看入口和展示容器。
