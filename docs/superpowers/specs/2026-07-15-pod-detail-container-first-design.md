# Pod 详情页 Container-first 改版设计

## 背景

当前 Pod 详情页使用超大资源名称、分散的操作按钮、独立摘要卡、等权 Tab 和多层嵌套卡片。
首屏重复展示 Node、Pod IP、重启等信息，Conditions 又使用与 Runtime 已确认浅色体系冲突的深色
表头。页面虽然包含必要字段，但无法帮助用户快速判断健康状态、定位具体 Container，并从同一上下文
进入日志或终端。

本次改版以用户确认的第三版视觉方案为基准，参考成熟 Kubernetes 控制台的资源详情模式，但继续使用
Aegis 现有浅色设计语言和 Project/Application/Environment 治理边界。

## 信息架构

详情页按三个层级组织：

1. 紧凑资源头：面包屑、Pod 名称、Running 状态、Application、Environment，以及刷新和更多操作。
2. 资源导航：概览、容器、事件、日志、YAML。概览承担健康判断与常用排障入口，其余页面承载专项任务。
3. 详情工作区：左侧为主要排障内容，右侧为稳定的 Pod 信息栏。

终端不再作为页面头部的等权按钮。用户在具体 Container 行选择“打开终端”，确保操作对象明确。删除
Pod 收进更多操作菜单并保留危险色、二次确认和生产环境强化文案。

## 概览页面

左侧首先展示 Container 列表。每个 Container 使用紧凑资源行而不是独立大卡片，显示名称、镜像、
运行状态、Ready、重启次数和启动时间。常用动作“查看日志”和“打开终端”位于对应 Container 行，
从该行进入时自动选中该 Container。

Container 列表下方依次展示 Conditions 和最近 Events。两者使用浅灰蓝表头、细分隔线和紧凑行高，
不使用黑色表头，也不再嵌套在额外的 surface 卡片中。布尔状态以小型状态标识呈现，异常信息优先使用
警示色，但正常状态不使用大面积绿色背景。

右侧 Pod 信息栏使用定义列表展示 Namespace、Node、Pod IP、Host IP、QoS Class、创建时间、启动时间
和累计重启。信息栏在桌面端保持窄列并可随页面滚动保持可见；窄屏时移动到主内容下方。

## 专项页面

- 容器：展示更完整的 Container 状态、镜像、Waiting/Terminated 原因和消息，并提供日志、终端入口。
- 事件：展示时间、类型、Reason、次数和消息，保持浅色紧凑表格。
- 日志：Container 和 Tail 行数选择位于代码区上方，刷新与复制为次级操作。
- YAML：保留完整只读资源内容和刷新操作。

现有 Runtime API、日志、YAML、删除 Pod 和 Exec 会话协议不变，本次仅重组前端信息架构和交互入口。

## 视觉规则

- Pod 名称桌面端不超过 24px，并允许长名称安全换行或省略。
- 正文使用现有产品字体和 13–14px 可读字号；控件高度保持约 32px。
- 间距使用 8、12、16、24px 节奏；优先用留白和分隔线建立层级。
- 不使用深色表头、巨型指标卡、大面积阴影、渐变或卡片套卡片。
- 状态、链接和危险操作是主要颜色强调，其余文本保持清晰的中性色对比。

## 组件边界

- `PodDetail.vue` 负责页面导航、数据加载、专项页面编排和受治理操作。
- `PodOverview.vue` 负责 Container-first 概览和右侧 Pod 信息展示，不发起 HTTP 请求。
- Container 行通过事件通知宿主打开指定 Container 的日志或终端。
- 现有 `RuntimeTerminalDrawer.vue`、Runtime API 和 view-model 继续复用。

## 错误与响应式

详情读取失败继续显示页面级错误；日志或 YAML 读取失败只影响对应专项页面。Container、Conditions 或
Events 为空时显示明确空状态，而不是留白。

桌面端使用主内容加约 300px 信息栏；低于 960px 时改为单列。操作入口保持可访问名称，键盘焦点清晰，
颜色不作为状态的唯一表达。

## 验收条件

- 首屏不再出现超大 Pod 名称、独立大摘要卡或深色 Conditions 表头。
- 概览以 Container 为第一排障对象，单行可进入该 Container 的日志和终端。
- Pod 基础信息集中在右侧定义列表，主内容不重复展示相同字段。
- 删除 Pod 只在更多菜单中出现，并继续执行现有确认与审计约束。
- 概览、容器、事件、日志和 YAML 的现有能力全部保留。
- 1440px 桌面和窄屏布局无溢出、遮挡或不可读文本。
- Runtime 相关自动化检查、前端类型检查和生产构建通过。

## 非目标

- 不新增 CPU、Memory 等当前 API 不提供的指标。
- 不新增监控图表、诊断服务、标签编辑或 Kubernetes 资源写入能力。
- 不改变后端 API、终端票据协议、审计内容或权限模型。
