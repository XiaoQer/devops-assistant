# Application Delivery Overview 与工作区统一设计

## 背景

Application、CI/CD 与 Runtime 当前分别演进，页面头部、字体层级、摘要卡片、间距和状态表达不一致。
现有 Overview 信息偏向活动堆叠，缺少围绕交付决策的应用健康、构建/部署状态、环境分布和待处理事项。

## 目标

- 将 Overview 重构为以交付总览为核心的项目驾驶入口。
- 统一 Application、CI/CD、Runtime 的页面头部、排版、卡片和间距基线。
- 优先复用已有 API 和数据，不新增虚假指标，不改变现有交付、日志和运行态能力。
- 保持页面在桌面和窄屏下可读，减少重复标题和过重装饰。

## 范围

### Overview 内容

- 统一页面头部：眉标、标题、描述和右侧动作。
- 顶部指标：Applications、Healthy、In delivery、Needs attention。
- 左侧“交付状态”：应用最近构建、当前环境目标状态和失败原因。
- 右侧“待处理事项”：失败流水线、待审批发布、环境/集群/Registry 配置异常。
- 环境分布：按环境展示应用覆盖和健康状态，数据来自已有应用环境与运行状态。
- 最近活动：复用已有 Pipeline/Release 活动，按时间排序展示。

### 统一工作区

- 抽取轻量共享页面基线，统一标题层级、描述、内容最大宽度、卡片边界、圆角、间距和状态颜色。
- Application、CI/CD、Runtime 接入同一套页面头部和摘要样式。
- 不要求三页内容完全相同；只统一视觉语法和信息密度。

## 非目标

- 不新增后端 API、数据库字段或 Kubernetes 查询。
- 不移除 Application 内发布记录、Pipeline 日志、Runtime 日志和 Pod 详情能力。
- 不伪造 CPU、内存、趋势或监控数据。
- 不在本次工作中重做整个全局 Design System。

## 验收条件

- [ ] Overview 首屏能直接回答：当前有哪些应用、多少健康、哪些交付中、哪些需要处理。
- [ ] Overview 同时展示交付状态、待处理事项、环境分布和最近活动四类信息。
- [ ] Overview 所有统计均来自现有真实数据，空数据有明确空状态。
- [ ] Application、CI/CD、Runtime 页面使用一致的页面头部高度、标题字号、描述字号和内容间距。
- [ ] 三个页面的状态颜色、卡片边界和操作按钮语义一致。
- [ ] 桌面端布局紧凑，窄屏下不出现横向溢出或信息重叠。
- [ ] 现有 Application、CI/CD、Runtime 的路由、日志、发布和运行态操作不回归。
- [ ] 前端测试、类型检查、生产构建和 `./scripts/verify.sh` 通过。

## 实现约束

- 前端请求继续集中在 `frontend/src/api`，View 不新增直接 HTTP 请求。
- 优先通过共享组件或共享样式变量实现统一，不复制三套近似 CSS。
- 现有用户未提交的 Portal 相关修改不纳入本次提交。
